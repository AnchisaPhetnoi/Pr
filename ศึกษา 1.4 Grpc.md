## 1.4  Grpc


####  gRPC คืออะไร? เข้าใจง่ายๆ

gRPC (Google Remote Procedure Call) คือเฟรมเวิร์กสำหรับการสื่อสารระหว่างบริการ (service) ต่างๆ ในระบบซอฟต์แวร์ โดยเฉพาะอย่างยิ่งในระบบไมโครเซอร์วิส (microservices) ที่มีการแบ่งงานย่อยออกเป็นส่วนเล็กๆ หลายส่วน

#### ทำไมต้อง gRPC?

    ประสิทธิภาพสูง: gRPC ใช้ HTTP/2 เป็นโปรโตคอลหลัก ซึ่งมีคุณสมบัติเด่นในการลดความหน่วง (latency) และเพิ่มประสิทธิภาพในการส่งข้อมูล ทำให้เหมาะกับระบบที่ต้องการความเร็วในการตอบสนอง
    ภาษาและแพลตฟอร์มที่หลากหลาย: gRPC รองรับการใช้งานกับหลากหลายภาษาโปรแกรม เช่น Go, Java, Python, C# ทำให้สามารถนำไปใช้ในระบบที่ใช้เทคโนโลยีที่แตกต่างกันได้
    กำหนดโครงสร้างข้อมูลได้อย่างชัดเจน: gRPC ใช้ Protobuf (Protocol Buffers) ในการกำหนดโครงสร้างข้อมูล ทำให้ทั้งไคลเอนต์และเซิร์ฟเวอร์เข้าใจข้อมูลที่ส่งหากันได้อย่างชัดเจน ลดโอกาสเกิดข้อผิดพลาด
    รองรับการสตรีมมิ่ง: gRPC รองรับการสตรีมมิ่งข้อมูลแบบสองทิศทาง (bidirectional streaming) ทำให้สามารถส่งข้อมูลขนาดใหญ่หรือข้อมูลแบบเรียลไทม์ได้อย่างมีประสิทธิภาพ

เปรียบเทียบ gRPC กับ REST
คุณสมบัติ	gRPC	REST
โปรโตคอล	HTTP/2	HTTP/1.1
ภาษาในการกำหนดโครงสร้างข้อมูล	Protobuf	JSON
ประสิทธิภาพ	สูง	ปานกลาง
การสตรีมมิ่ง	รองรับ	ไม่รองรับโดยตรง

#### สรุป

gRPC เป็นตัวเลือกที่ดีสำหรับการสร้างระบบไมโครเซอร์วิสที่มีประสิทธิภาพสูงและต้องการความยืดหยุ่นในการปรับเปลี่ยน โดยเฉพาะอย่างยิ่งในกรณีที่ต้องการสื่อสารข้อมูลขนาดใหญ่หรือข้อมูลแบบเรียลไทม์ อย่างไรก็ตาม การเลือกใช้ gRPC หรือ REST จะขึ้นอยู่กับความเหมาะสมของแต่ละโครงการ

#### อยากรู้เพิ่มเติมเกี่ยวกับ gRPC?

    ข้อดี: เหมาะสำหรับระบบที่ต้องการประสิทธิภาพสูง, รองรับหลากหลายภาษาและแพลตฟอร์ม, กำหนดโครงสร้างข้อมูลได้อย่างชัดเจน
    ข้อเสีย: อาจมีความซับซ้อนในการเรียนรู้และใช้งานมากกว่า REST, ต้องใช้ Protobuf ซึ่งอาจเป็นสิ่งใหม่สำหรับบางคน
    กรณีศึกษา: มักใช้ในระบบไมโครเซอร์วิสขนาดใหญ่, ระบบที่ต้องการประสิทธิภาพสูง เช่น ระบบเกมออนไลน์, ระบบ IoT

#### คำแนะนำ:

หากคุณกำลังพิจารณาจะนำ gRPC ไปใช้งานในโครงการของคุณ ลองศึกษาเพิ่มเติมเกี่ยวกับ Protobuf และ HTTP/2 เพื่อทำความเข้าใจหลักการทำงานของ gRPC ได้อย่างลึกซึ้งยิ่งขึ้น

#### คำสำคัญ: gRPC, RPC, microservices, HTTP/2, Protobuf, REST

#### หมายเหตุ: ข้อมูลนี้เป็นเพียงภาพรวมเบื้องต้นของ gRPC หากต้องการข้อมูลเพิ่มเติม สามารถค้นหาข้อมูลเพิ่มเติมได้จากแหล่งข้อมูลต่างๆ เช่น เอกสารทางการของ gRPC หรือบทความที่เกี่ยวข้อง

## gRPC
    กูเกิลพัฒนาระบบเรียกฟังก์ชั่นการทำงานข้ามเครื่อง (remote procedure call - RPC) ของตัวเองชื่อ gRPC มาได้สักพัก และในที่สุดก็ออกเวอร์ชัน 1.0 แล้ว

    gRPC เป็นเฟรมเวิร์คสำหรับเรียกคำสั่งข้ามเครื่องที่ออกแบบให้ใช้งานในยุคอินเทอร์เน็ต รองรับการเรียกคำสั่งเป็นหลัก 'หมื่นล้าน' คำสั่งต่อวินาที หน้าที่ของมันคือเป็นสื่อกลางระหว่างระบบต่างๆ ให้คุยกันอย่างอัตโนมัติ รองรับภาษาที่หลากหลายทั้ง C++, Java, PHP, Go, Node, Ruby, Python, C# รวมถึงทำงานได้ทุกแพลตฟอร์มสำคัญ Windows, Mac, Linux, Android, iOS

    gRPC ถูกออกแบบมาใช้กับโพรโทคอล HTTP/2 และใช้แทนวิธีการสื่อสารของบางระบบที่ส่งข้อมูลในฟอร์แมต JSON บน HTTP ข้อดีที่เหนือกว่าคือมันเร็วกว่าและมีประสิทธิภาพมากกว่า ตัวอย่างองค์กรที่ใช้ gRPC นอกจากกูเกิลแล้วยังมี Netflix, Core OS, Cisco, Juniper, Arista, Square โดยตัวอย่างการใช้งาน gRPC ของ Square

บทความนี้มีวัตถุประสงค์เพื่อให้มีแนวคิดระดับสูงเกี่ยวกับ gRPC นอกจากนี้ยังจะอธิบายถึงความเหมือนและความแตกต่างระหว่าง gRPC และโปรโตคอลและสถาปัตยกรรมที่มีอยู่ตามมาสำหรับการสื่อสารของเว็บแอปพลิเคชัน

      gRPC เป็นเฟรมเวิร์ก Remote Procedure Call แบบโอเพนซอร์สที่ใช้สำหรับการสื่อสารที่มีประสิทธิภาพสูงระหว่างบริการ เป็นวิธีที่มีประสิทธิภาพในการเชื่อมต่อบริการที่เขียนในภาษาต่างๆด้วยการสนับสนุนแบบเสียบได้สำหรับการทำโหลดบาลานซ์การติดตามการตรวจสอบความสมบูรณ์และการพิสูจน์ตัวตน โดยค่าเริ่มต้น gRPC จะใช้บัฟเฟอร์โปรโตคอลสำหรับการทำให้ข้อมูลที่มีโครงสร้างเป็นอนุกรม โดยทั่วไป gRPC ถือเป็นทางเลือกที่ดีกว่าสำหรับโปรโตคอล REST สำหรับสถาปัตยกรรมไมโครเซอร์วิส "g" ใน gRPC สามารถระบุแหล่งที่มาของ Google ซึ่งเป็นผู้พัฒนาเทคโนโลยีในตอนแรก
    ก่อนที่จะลงรายละเอียดเพิ่มเติมเกี่ยวกับ gRPC เรามาดูสถาปัตยกรรมไมโครเซอร์วิสกันก่อน


## Microservices กับ Monoliths

สถาปัตยกรรมเสาหินเป็นวิธีดั้งเดิมในการออกแบบแอปพลิเคชัน ประกอบด้วยโค้ดเบสเดียวที่แยกไม่ออกซึ่งให้บริการอินเทอร์เฟซผู้ใช้ฝั่งไคลเอ็นต์แอปพลิเคชันฝั่งเซิร์ฟเวอร์และฐานข้อมูล นักพัฒนาทั้งหมดที่ทำงานในโปรเจ็กต์จะส่งรหัสไปยังที่เก็บเดียวกัน หนึ่งในการเปรียบเทียบที่ฉันชอบเกี่ยวกับเสาหินคือการคิดว่ามันเป็นสตูดิโออพาร์ตเมนต์ ห้องเดี่ยวจะถูกแบ่งออกเป็นช่องต่างๆตามความต้องการ

ข้อดีของสถาปัตยกรรมเสาหินคือเนื่องจากมีเพียงหน่วยเดียวการดำเนินการเช่นการบันทึกการตรวจสอบประสิทธิภาพและการแคชจึงทำได้อย่างง่ายดาย นอกจากนี้ยังง่ายต่อการพัฒนาทดสอบแก้ไขข้อบกพร่องและปรับใช้

แต่เมื่อแอปพลิเคชันเติบโตขึ้นการดูแลรักษาปรับขนาดและทำความเข้าใจก็ยากขึ้น นอกจากนี้อาจมีความซับซ้อนมากจนการเปลี่ยนแปลงโค้ดเพียงเล็กน้อยอาจส่งผลต่อแอปพลิเคชันทั้งหมด

ข้อเสียที่สำคัญอีกประการหนึ่งของเสาหินก็คือความมุ่งมั่นที่เหนียวแน่นต่อเทคโนโลยีเดียว การนำเฟรมเวิร์กหรือภาษาใหม่มาใช้อาจจำเป็นต้องเขียนระบบใหม่ทั้งหมด

## เข้าสู่สถาปัตยกรรมไมโครเซอร์วิส!

หากสถาปัตยกรรมเสาหินเป็นอพาร์ทเมนต์สตูดิโอสถาปัตยกรรมไมโครเซอร์วิสถือได้ว่าเป็นบ้านที่มีหลายห้อง นั่นหมายความว่าแอปพลิเคชันทั้งหมดจะถูกแบ่งย่อยออกเป็นแอปพลิเคชันหรือบริการขนาดเล็กหลายรายการ

สิ่งนี้ช่วยให้ทีมนักพัฒนามีความยืดหยุ่นในการเลือกเทคโนโลยีที่เหมาะสมกับความต้องการมากที่สุดและสามารถให้พวกเขาปรับขนาดบริการได้อย่างอิสระ ข้อผิดพลาดใด ๆ ในแอปพลิเคชันไมโครเซอร์วิสจะส่งผลต่อบริการบางอย่างเท่านั้นไม่ใช่ทั้งแอปพลิเคชัน

บริการเหล่านี้สามารถพัฒนาดูแลและใช้งานได้โดยอิสระและสื่อสารกันผ่านวิธีการที่กำหนดไว้ซึ่งเรียกว่า API (Application Programming Interfaces)

การสื่อสารระหว่างไมโครเซอร์วิสผ่าน HTTP สามารถทำได้หลายวิธี วิธีที่ใช้กันอย่างแพร่หลายคือปฏิบัติตามโปรโตคอล REST gRPC เป็นอีกวิธีหนึ่งในการสื่อสารนี้ สร้างขึ้นเพื่อเอาชนะข้อ จำกัด ของ REST ในการสื่อสารด้วยไมโครเซอร์วิส
สถาปัตยกรรม REST

REST เป็นสถาปัตยกรรมเว็บที่ใช้โปรโตคอล HTTP มีการใช้กันอย่างแพร่หลายในการพัฒนาเว็บแอปพลิเคชัน พูดง่ายๆก็คือ REST คือความสัมพันธ์ไคลเอนต์เซิร์ฟเวอร์ที่ข้อมูลส่วนหลังถูกสร้างขึ้นผ่านการแสดงอย่างง่ายเช่น JSON / XML ไปยังไคลเอนต์ REST ย่อมาจาก REpresentational State Transfer ตามที่ Roy Fielding อธิบาย REST เป็นโปรโตคอลที่ไม่บังคับใช้กฎเกณฑ์ใด ๆ เกี่ยวกับวิธีการนำไปใช้ในระดับที่ต่ำกว่า เป็นแนวทางสำหรับการใช้งานสถาปัตยกรรมระดับสูง

## ในการทำให้แอปพลิเคชันใด ๆ พักผ่อนอย่างแท้จริงต้องปฏิบัติตามข้อ จำกัด ทางสถาปัตยกรรมหกประการ:

    อินเทอร์เฟซที่เหมือนกัน: ความหมายอินเตอร์เฟส API ต้องแสดงต่อทรัพยากรในเว็บแอปพลิเคชันต่อผู้บริโภคของ API
    ไคลเอนต์เซิร์ฟเวอร์: ไคลเอนต์และเซิร์ฟเวอร์ต้องเป็นอิสระจากกันและไคลเอนต์ควรทราบเฉพาะ URI ของทรัพยากร
    ไม่มีสถานะ: เซิร์ฟเวอร์ต้องไม่จัดเก็บสิ่งที่เกี่ยวข้องกับคำขอของไคลเอ็นต์ ลูกค้ามีหน้าที่รับผิดชอบในการรักษาสถานะของแอปพลิเคชัน
    แคชได้: ทรัพยากรต้องสามารถแคชได้
    ระบบเลเยอร์: สถาปัตยกรรมต้องเป็นเลเยอร์ซึ่งหมายความว่าส่วนประกอบของสถาปัตยกรรมสามารถอยู่ในเซิร์ฟเวอร์หลายเครื่องได้
    รหัสตามความต้องการ: ไคลเอนต์ต้องสามารถรับโค้ดที่ปฏิบัติการได้เป็นการตอบสนอง นี่เป็นข้อ จำกัด ที่เป็นทางเลือก

    GET- การเข้าถึงทรัพยากรแบบอ่านอย่างเดียว
    POST- สร้างทรัพยากรใหม่
    DELETE- ลบทรัพยากร
    PUT- อัปเดตทรัพยากรที่มีอยู่ / สร้างทรัพยากรใหม่

RPC ย่อมาจาก Remote Procedure Call ตามชื่อที่แนะนำแนวคิดก็คือเราสามารถเรียกใช้ฟังก์ชัน / วิธีการบนเซิร์ฟเวอร์ระยะไกลได้ โปรโตคอล RPC ช่วยให้สามารถรับผลลัพธ์ของปัญหาในรูปแบบเดียวกันไม่ว่าจะดำเนินการที่ใด อาจเป็นแบบโลคัลหรือในเซิร์ฟเวอร์ระยะไกลโดยใช้ทรัพยากรที่ดีกว่า

RPC เป็นโปรโตคอลที่เก่ากว่า REST มาก ถูกใช้มาตั้งแต่สมัยของ ARPANET ในปี 1970 เพื่อดำเนินการเครือข่าย คำว่า RPC ได้รับการประกาศเกียรติคุณเป็นครั้งแรกโดย Bruce Jay Nelson ในปี 1981 แต่อย่างที่เรากำลังจะเห็น RPC ยังคงมีความเกี่ยวข้องและนำไปใช้ในแอปพลิเคชันสมัยใหม่ที่ใช้ API ในรูปแบบต่างๆ

ความคิดก็เหมือนกัน API สร้างขึ้นโดยการกำหนดวิธีการสาธารณะ จากนั้นเมธอดจะถูกเรียกด้วยอาร์กิวเมนต์ RPC เป็นเพียงฟังก์ชันจำนวนมาก แต่ในบริบทของ HTTP API จะเป็นการใส่เมธอดใน URL และอาร์กิวเมนต์ในสตริงหรือเนื้อความของแบบสอบถาม

APIs RPC จะใช้สิ่งที่ชอบPOST /deleteResourceกับร่างกายของแทนวิธีการส่วนที่เหลือซึ่งจะเป็น{ “id”: 1 }DELETE /resource/1

RPC เป็นที่นิยมอย่างมากสำหรับอุปกรณ์ IoT และโซลูชันอื่น ๆ ที่ต้องการการสื่อสารตามสัญญาที่กำหนดเองสำหรับอุปกรณ์ที่ใช้พลังงานต่ำเนื่องจากการดำเนินการคำนวณส่วนใหญ่สามารถถ่ายโอนไปยังอุปกรณ์อื่นได้ ตามเนื้อผ้า RPC สามารถนำไปใช้เป็น RPC-XML และ RPC-JSON

gRPC เป็นเฟรมเวิร์กล่าสุดที่สร้างบนโปรโตคอล RPC ใช้ประโยชน์จากข้อดีและพยายามแก้ไขปัญหาของ RPC แบบดั้งเดิม
gRPC อีกครั้งคืออะไร?

จากสิ่งที่เราอ่านจนถึงตอนนี้เราสามารถกำหนด gRPC ใหม่ได้ เป็นการปรับใช้กรอบ RPC แบบเดิม แล้วอะไรที่ทำให้มันแตกต่างจากเฟรมเวิร์ก RPC ที่มีอยู่?

ความแตกต่างที่สำคัญที่สุดคือ gRPC ใช้โปรโตคอลบัฟเฟอร์เป็นภาษานิยามอินเทอร์เฟซสำหรับการทำให้เป็นอนุกรมและการสื่อสารแทน JSON / XML บัฟเฟอร์โปรโตคอลสามารถอธิบายโครงสร้างของข้อมูลและโค้ดสามารถสร้างขึ้นจากคำอธิบายนั้นเพื่อสร้างหรือแยกวิเคราะห์สตรีมไบต์ที่แสดงถึงข้อมูลที่มีโครงสร้าง นี่คือเหตุผลที่ gRPC เป็นที่ต้องการสำหรับเว็บแอปพลิเคชันที่มีหลายภาษา (ใช้กับเทคโนโลยีที่แตกต่างกัน) รูปแบบข้อมูลไบนารีช่วยให้การสื่อสารเบาลง gRPC ยังสามารถใช้กับรูปแบบข้อมูลอื่น ๆ ได้ แต่รูปแบบที่ต้องการคือบัฟเฟอร์โปรโตคอล หากต้องการทราบข้อมูลเพิ่มเติมเกี่ยวกับบัฟเฟอร์โปรโตคอลในรายละเอียดตรวจสอบนี้บทความ

นอกจากนี้ gRPC ยังสร้างขึ้นบน HTTP / 2 ซึ่งรองรับการสื่อสารแบบสองทิศทางพร้อมกับคำขอ / การตอบกลับแบบเดิม gRPC อนุญาตให้มีการเชื่อมต่อแบบหลวม ๆ ระหว่างเซิร์ฟเวอร์และไคลเอนต์ ในทางปฏิบัติไคลเอ็นต์จะเปิดการเชื่อมต่อที่ใช้งานได้ยาวนานกับเซิร์ฟเวอร์ gRPC และสตรีม HTTP / 2 ใหม่จะเปิดขึ้นสำหรับการเรียก RPC แต่ละครั้ง
REST กับ gRPC

ซึ่งแตกต่างจาก REST ซึ่งใช้ JSON (ส่วนใหญ่) gRPC ใช้โปรโตคอลบัฟเฟอร์ซึ่งเป็นวิธีการเข้ารหัสข้อมูลที่ดีกว่า เนื่องจาก JSON เป็นรูปแบบข้อความจึงมีน้ำหนักมากกว่าข้อมูลที่บีบอัดในรูปแบบ protobuf มาก

การปรับปรุง gRPC ที่สำคัญอีกประการหนึ่งจาก REST ทั่วไปคือใช้ HTTP 2 เป็นโปรโตคอลการถ่ายโอน HTTP 1.1 ซึ่งส่วนใหญ่ใช้โดย REST เป็นแบบจำลองการตอบสนองการร้องขอ (REST สามารถใช้กับ HTTP2 ได้เช่นกัน) gRPC ใช้คุณลักษณะการสื่อสารแบบสองทิศทางของ HTTP 2 พร้อมกับโครงสร้างคำขอตอบกลับแบบเดิม ใน HTTP 1.1 เมื่อมีการร้องขอหลายรายการจากไคลเอนต์หลายรายการระบบจะให้บริการทีละรายการ ซึ่งอาจทำให้ระบบทำงานช้าลง HTTP 2 อนุญาตการมัลติเพล็กซ์ดังนั้นจึงสามารถให้บริการคำขอและการตอบกลับหลายรายการในเวลาเดียวกันได้

เราสามารถสรุปได้ว่า gRPC เป็นตัวเลือกที่ยอดเยี่ยมเมื่อกรณีการใช้งานเกี่ยวข้องกับการสื่อสารหลายภาษาด้วย idiomatic API หรือการสื่อสารไมโครเซอร์วิสขนาดใหญ่
